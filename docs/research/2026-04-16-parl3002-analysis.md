# PARL3002 Cyclomatic Complexity Analysis

**Date:** 2026-04-16
**Status:** Internal metric only. PARL3002 analyzer deferred. Spike passed; CFG predicate-counting is the shipped implementation.
**Current implementation:** `src/Parlance.CSharp.Analyzers/Metrics/Experimental/CyclomaticComplexityMetric.cs` — CFG-based predicate counter, single path, no syntactic fallback. The metric is shipped under the `Experimental` namespace to signal that it is consumer-facing only (PARL3101, future curation), not a published API. See the Post-Spike Decision Log below for the full rationale.

---

## Post-Spike Decision Log (2026-04-16)

**Spike result:** Pass. CFG predicate-counting scored `ManySequentialIfs` at 84, matching the upstream gold value. All other upstream fixtures (`SomeComplexMethod` → 12, `ManyDeclarations` → 1, `BoolAssignments` → 1) also matched. Implementation proceeded with CFG as the primary path.

**Fallback decision: removed.** The first refactor pass added a syntax-based decision walker as a fallback for when `GetOperation` returned null or `ControlFlowGraph.Create` rejected the operation. During review (see the complexity-implementation review on `feature/curation-start`, § "Cyclomatic fallback can return a different metric value for the same code") this was rejected for two reasons:

1. **Path-dependent numbers.** The syntax walker and the CFG predicate counter produce different values for the same code in at least one documented shape — a `switch` expression with a discard arm scores 3 via syntax and 4 via CFG (the CFG models the implicit "no match throws" branch that the discard syntactically swallows). A threshold-based analyzer (PARL3002 at, say, 10) would fire or not fire on the same method depending on whether Roslyn happened to produce a usable `IOperation`. That is the class of bug the whole refactor was supposed to eliminate — silently giving a plausible-looking wrong answer when the defensible answer is unavailable.

2. **No real trigger in practice.** The "semantic model is null" case did not have any actual caller in Parlance; making the parameter optional was API-dressing, not a real use case. The "CFG decline" cases (`GetOperation` returning null, non-root operation shapes, `ArgumentException` from `ControlFlowGraph.Create`) are rare-to-hypothetical and not worth a second algorithm.

**Final contract:**
- `SemanticModel` is required.
- CFG is the only calculation path.
- When CFG cannot be built, return a `Skipped` result with a reason string naming the specific failure (no `IOperation`, non-root operation, or `ArgumentException`). Threshold consumers must check `SkippedReason` before using `Complexity`.
- No syntactic approximation. No file-scoped diagnostics guard. Roslyn's semantic analysis is trusted to decide whether a CFG can be produced; we do not second-guess it with a syntax walker.

**Philosophy:** an honest skip is better than a path-dependent number. This follows the review doc's preferred fix.

## Why This Was Called "Difficult"

The original plan (spec Task 6) wanted a Roslyn-native McCabe metric using `ControlFlowGraph` with the standard formula: `E - N + 2` (edges minus nodes plus 2). This failed because:

**Roslyn's CFG optimizes away empty branches.** For `if (t != null) { }`, ReSharper's PSI-based CFG creates distinct true/false edges even though the true-branch body is empty. Roslyn's `ControlFlowGraph` collapses these — both the true and false paths lead to the same continuation block, so the edge/node difference disappears. The `ManySequentialIfs` fixture (83 empty `if` statements) scored 84 in ReSharper but 1 via Roslyn's `E - N + 2`.

The contract documents this: "Roslyn CFG collapsed the upstream empty-branch stress fixture (ManySequentialIfs) to complexity 1, so CFG parity was not feasible without additional interpretation."

The implementation fell back to syntax-based counting, which produces correct numbers but abandons the graph-theoretic foundation.

## Bugs in the Pre-Spike Implementation (resolved)

The items below describe bugs in the v1 syntax-based walker that prompted the spike. **All are resolved in the current CFG-based implementation** — they are retained here as a reviewer's record of what the refactor had to fix, not as outstanding work.

### Critical: `GetDiagnostics().Any()` is file-scoped, not body-scoped

```csharp
// Line 29 of CyclomaticComplexityMetric.cs
if (bodyOrExpression is null || bodyOrExpression.SyntaxTree.GetDiagnostics().Any())
    return Skipped("No complete supported body was available.");
```

`SyntaxTree.GetDiagnostics()` returns **parse diagnostics for the entire file**, not just the method body. If ANY method in the file has a parse error (missing brace, missing semicolon), EVERY method in that file returns `Skipped`. This is catastrophic for real-world use.

**Fix options:**
- Use `bodyOrExpression.ContainsDiagnostics` — checks only the node and its descendants
- Remove the check entirely and calculate best-effort (matching the cognitive metric's philosophy)
- Check `bodyOrExpression.GetDiagnostics()` for the specific span

### Important: `EdgeCount` and `NodeCount` are fake values

```csharp
return new CyclomaticComplexityResult(
    complexity,
    edgeCount: complexity,    // just echoes complexity
    nodeCount: 1);            // always 1
```

The result object promises graph-theoretic data (`EdgeCount`, `NodeCount`) but delivers fabricated values. Any consumer relying on these would get wrong answers. The contract acknowledges this ("must not be used for a visible analyzer decision until CFG accounting is revisited") but the values should be nullable or the fields should be removed, not populated with misleading numbers.

### Important: `SemanticModel` parameter is required but never used

```csharp
public static CyclomaticComplexityResult Calculate(SyntaxNode declaration, SemanticModel semanticModel)
```

The `Walker` class has no reference to the semantic model. This is a placeholder for a future CFG approach but currently forces callers to provide a compilation for nothing. Should be optional (`SemanticModel? semanticModel = null`).

### Important: `catch when` filters not counted

The walker counts `catch` clauses (+1 each) but ignores `when` filters on catch clauses. Exception filter `when` clauses are decision points in McCabe counting — `CA1502` counts them. This is an under-count.

### Important: Switch `when` guards not counted

Both switch statement `case X when Y:` and switch expression `X when Y =>` — the `when` guard adds a decision point beyond the base case/arm. Neither is counted. The walker counts non-default labels and non-discard arms, but not their `when` guards.

### Minor: `CyclomaticComplexityResult` is a class, not a positional record

Project convention says "positional record syntax / primary constructors." The result type uses a manual constructor.

### Minor: API inconsistency with `CognitiveComplexityMetric`

| Aspect | Cognitive | Cyclomatic |
|---|---|---|
| Input | `bodyOrExpression` (body node) | `declaration` (whole declaration) |
| Body extraction | Caller's responsibility | Internal to `Calculate` |
| `SemanticModel` | Optional (`SemanticModel?`) | Required (`SemanticModel`) |
| Return type | `int` | `CyclomaticComplexityResult` |
| Visibility | `public static` | `public static` |

Different callers need different setup code for the same kind of metric.

### Minor: Duplicate test case

`ComplexMethod_DefaultSettingsMethod` and `ComplexMethod_ModifiedThresholdMethodScore` are identical calls — same source, same method name, same expected score (12). The second exists for port-matrix traceability but provides zero test value.

## Why the ReSharper Approach Can't Be Ported Directly

The upstream `ComplexityAnalysisElementProblemAnalyzer.cs` uses ReSharper-specific APIs:

1. **`IControlFlowBuilder`** — ReSharper's PSI-based control flow graph builder. No Roslyn equivalent with identical behavior.

2. **`EdgeEqualityComparer`** — ReSharper's C# graph treats boolean values as conditions, producing duplicate edges from one node to the next. The comparer deduplicates by source+target. Roslyn's CFG doesn't have this behavior.

3. **`FudgeGraph` / unary `!` workaround** — ReSharper treats `!` as a conditional, creating false branch edges. The code walks backward through the graph to remove these fake paths. Roslyn doesn't treat `!` as a branch, so this workaround is unnecessary.

4. **Class-level aggregation** — The plugin reports complexity at both class and method level. Gold files show `ComplexityExample` with complexity 1 at class level. Parlance is method/accessor-scoped only.

In short: the ReSharper implementation is working around ReSharper-specific CFG quirks that don't exist in Roslyn. Porting the workarounds would be porting bugs.

## Spike: Roslyn CFG Predicate-Counting

### Hypothesis

The original CFG failure used `E - N + 2`. But McCabe has an equivalent definition:

> **Cyclomatic complexity = number of predicate nodes + 1**

A predicate node is a basic block with more than one successor. In Roslyn's `ControlFlowGraph`, this is any `BasicBlock` where `ConditionalSuccessor` is non-null.

Even when Roslyn collapses empty `if` bodies (making both successors point to the same block), the `ConditionalSuccessor` should still be present because the branch instruction exists in the syntax. This means predicate-counting should give 84 for `ManySequentialIfs` even though `E - N + 2` gives 1.

### Feasibility

**Confirmed:** `ControlFlowGraph`, `BasicBlock`, `ControlFlowBranch`, and `ConditionalSuccessor` are all present in the `Microsoft.CodeAnalysis.Common 4.12.0` netstandard2.0 assembly. The analyzers project references this version already. No new dependencies needed.

### Spike Test

Write a single test that:
1. Compiles the `ManySequentialIfs` source (83 empty `if` statements)
2. Gets `IOperation` from `SemanticModel.GetOperation(bodyNode)`
3. Builds `ControlFlowGraph.Create(operation)`
4. Counts basic blocks where `ConditionalSuccessor is not null`
5. Asserts `predicateCount + 1 == 84`

### Implementation Sketch (if spike succeeds)

```csharp
var operation = semanticModel.GetOperation(bodyNode);
if (operation is null)
    return SyntaxFallback(bodyNode); // graceful degradation

var cfg = ControlFlowGraph.Create(operation);

int predicateCount = 0;
int edges = 0;
int nodes = 0;

foreach (var block in cfg.Blocks)
{
    if (block.Kind == BasicBlockKind.Block)
        nodes++;
    if (block.ConditionalSuccessor is not null)
    {
        predicateCount++;
        edges += 2; // true + false
    }
    else if (block.FallThroughSuccessor is not null)
    {
        edges += 1;
    }
}

return new CyclomaticComplexityResult(
    Complexity: predicateCount + 1,
    EdgeCount: edges,
    NodeCount: nodes);
```

## Approaches Considered (for reference)

### Approach A: Fix bugs in syntax-based counting

The syntax walker already produces correct scores. Fix the bugs listed above. Keep `1 + decision_count` formula.

**Pros:** Minimal change. Already matches all upstream fixtures. Reliable.
**Cons:** Can't provide real graph data. The `1 + decision_count` formula is technically a simplification of McCabe, not the real thing (they agree for structured code but diverge for `goto` spaghetti).

### Approach B: Predicate-counting on Roslyn CFG (the spike)

Use `ControlFlowGraph` and count basic blocks with non-null `ConditionalSuccessor` instead of `E - N + 2`.

**Pros:** Real graph-theoretic data. `EdgeCount`/`NodeCount` become meaningful. Matches the McCabe definition properly. Handles `goto`, unreachable code, and other non-structured patterns correctly.
**Cons:** Untested — the key hypothesis needs verification. Requires `IOperation` (semantic model), adding weight. Some syntax patterns may not support `GetOperation()` (e.g., incomplete code). Needs a syntax-based fallback.

### Approach C: Hybrid — syntax primary, CFG validation

Use syntax counting as the primary metric. Attempt CFG construction as an optional validator. If CFG succeeds and disagrees, attach a note to the result.

**Pros:** Best-of-both reliability.
**Cons:** Two code paths. When they disagree, which is right? Complexity for unclear benefit.

### Approach D: Ship PARL3002 as disabled-by-default / curation-only

Create `PARL3002_CyclomaticComplexityThreshold` with `isEnabledByDefault: false`. Activate only through curation sets.

**Differentiators from CA1502:**
1. **Source-level vs IL-level:** CA1502 operates on compiled IL. PARL3002 counts at syntax level.
2. **Curation integration:** PARL3002 participates in Parlance curation sets and AI-review workflows.
3. **PARL3101 substrate:** The metric feeds future test-adequacy analysis.
4. **Richer diagnostic properties:** Message can carry decision-count breakdown for AI consumers.

### Approach E: Remove PARL3002 entirely (if spike fails)

Remove the metric and tests. File a GitHub story with this research. Revisit when PARL3101 needs it.

---

## Relationship to CA1502

`CA1502` already reports excessive cyclomatic complexity. It operates at IL level. Parlance should not ship an enabled duplicate without documented distinction. Possible differentiators:

- Source-level counting sees switch expressions, pattern matching, expression-bodied members as the developer wrote them, not as the compiler emitted them
- Curation set integration
- PARL3101 test-adequacy substrate (Parlance needs to own the metric definition)
- AI-readable diagnostic properties

Until at least one of these is actively used, PARL3002 should remain disabled-by-default or absent.
