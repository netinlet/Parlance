# PARL3001 — Cognitive complexity exceeds threshold

| | |
|---|---|
| **ID** | `PARL3001` |
| **Category** | Maintainability |
| **Default severity** | Warning |
| **Code fix** | None — refactor manually |
| **Method threshold (default)** | `15` |
| **Property/accessor threshold (default)** | `3` |

## What it reports

`PARL3001` flags declarations whose control-flow nesting and interruption make
the code harder to follow. It scores cognitive complexity per declaration and
reports when the score exceeds the configured threshold.

The diagnostic is a quality gate — there is no automatic code fix. Resolve it
by extracting helpers, flattening nested branches, or simplifying boolean
expressions.

## Supported declarations

The analyzer scores method-shaped and accessor-shaped declarations:

- Methods, constructors, destructors, operators, and conversion operators.
- Property, indexer, and event accessors (`get` / `set` / `init` / `add` / `remove`).
- Property and indexer arrow bodies (`=>` expression-bodied members).
- Local functions (see scoping split below).
- Both block bodies and expression bodies.

Lambdas and anonymous methods contribute to the score of their containing
declaration; they are not reported independently.

### Local function scoping split

- **Non-static local functions** are folded into their containing
  declaration's score. Their body adds to the parent at `parent nesting + 1`,
  reflecting their shared-closure semantics.
- **Static local functions** are scored *independently*. The parent
  declaration's score excludes the static local's body, and `PARL3001`
  evaluates the static local against its own threshold.

This prevents double-counting code the author already isolated with `static`.

## Scoring rules

Each construct adds the listed amount to the declaration's score. `nesting` is
the number of enclosing decision/loop/lambda scopes at the increment site.

| Construct | Increment |
|---|---|
| `if` (new decision) | `+1 + nesting` |
| `else if` | `+1` (no extra nesting penalty) |
| `else` | `+1` |
| `for`, `foreach`, `while`, `do` | `+1 + nesting` |
| `switch` statement / switch expression | `+1 + nesting` (per switch, not per case/arm) |
| Conditional expression (`?:`) | `+1 + nesting` |
| `catch` | `+1 + nesting`; exception filter (`when`) adds `+1` |
| Boolean operator group (`&&` / `\|\|`) | `+1` per maximal run; alternation starts a new group |
| Pattern operator group (`and` / `or`) | `+1` per maximal run; alternation starts a new group |
| Recursion (direct self-call, semantic) | `+1` |
| `goto` (any form) | `+1 + nesting` |
| `break` | `+1` |
| `continue` | `0` |

Boolean and pattern operator groups are scored consistently: a logically
equivalent shape produces the same increment count whether it is written with
`&&`/`||` or `and`/`or`.

## Configuration

`PARL3001` exposes two thresholds — one for method-shaped declarations and a
smaller one for property-shaped declarations. Accessors are conventionally
trivial, so an equally complex accessor is a stronger smell than an equally
complex method.

```ini
# .editorconfig

# Method-shaped declarations (methods, constructors, destructors,
# operators, conversion operators, local functions). Default: 15.
dotnet_code_quality.PARL3001.max_cognitive_complexity = 15

# Property-shaped declarations (property/indexer arrow bodies and every
# accessor kind). Default: 3.
dotnet_code_quality.PARL3001.max_cognitive_complexity.property = 3
```

Invalid, missing, or non-positive values fall back to the shape-specific
default.

## Diagnostic shape

- **Message:** `'{Name}' has cognitive complexity {Score}, exceeding threshold {Threshold}`.
- **Primary location:** the declaration's identifier when available; otherwise
  the declaration node.
- **Additional locations:** one per increment, in walk order, anchored on the
  token that triggered the contribution (the `if` keyword, the `||` operator,
  the recursive call site, etc.).
- **Properties:** `increment.count` plus `increment.{N}.reason` for each
  additional location. Tooling (IDE hovers, MCP consumers) reads these to
  explain *why* the score crossed the threshold, not just *how far*.

## Generated and invalid code

Generated code is excluded from analysis. Incomplete or invalid syntax is
analyzed best-effort; the rule does not throw on missing bodies and produces
no diagnostic when the score cannot be computed reliably.

## Design notes

### Why `break` is `+1`

Cognitive complexity literature is split on `break`:

- The Sonar S3776 white paper scores `break` as `0`, treating it as a simple
  control transfer.
- The JetBrains Cognitive Complexity plugin scores it `+1`, treating it as a
  small interruption to linear reading.

Parlance scores `break` as `+1` (JetBrains parity). The value is the named
constant `ComplexityDefaults.BreakIncrement`; flipping the constant to `0`
switches behavior in one place if a project prefers Sonar parity.
